GNU GDB Security Policy
=======================

Introduction
------------

  The GNU Debugger (GDB) is a tool for diagnosing issues "inside"
  another program.  This is done by controlling the execution of the
  program being debugged and allowing the user to inspect and modify
  the state of the running program.  The program being debugged may be
  local i.e. on the system on which GDB runs or remote, on a different
  system.

Scope Of This Policy
--------------------

  This policy covers all currently supported versions of GDB as
  released from the official GDB website, and covers gdb, gdbserver,
  as well as any additional tools packaged within a GDB release.  The
  official GDB website, and the source of supported GDB releases can
  be found here:

    https://sourceware.org/gdb/

Policy Objectives
-----------------

  The objective of this policy is define what the GDB project
  considers a security bug and what is a non-security bug.  How each
  category of bug should be reported.  And to give guidance to users
  on how to use GDB in a secure fashion.

Reporting Public Security And Non-Security Bugs
-----------------------------------------------

  NOTE: All bugs reported in the GDB Bugzilla are public.

  It is expected that critical security bugs will be rare, and that
  most security bugs, along with all non-security bugs, can be
  reported in GDB Bugzilla system, thus making them public
  immediately.  The system can be found here:

    https://sourceware.org/bugzilla/

Reporting Private Security Bugs
-------------------------------

  In order to report a private security bug that is not immediately
  public, please contact one of the downstream distributions with
  security teams.  The following teams have volunteered to handle
  such bugs:

     Red Hat: secalert@redhat.com
     SUSE:    security@suse.de

  Please report the bug to just one of these teams.  It will be shared
  with other teams as necessary.
  The team contacted will take care of details such as vulnerability
  rating and CVE assignment (http://cve.mitre.org/about/).  It is likely
  that the team will ask to file a public bug because the issue is
  sufficiently minor and does not warrant an embargo.  An embargo is not
  a requirement for being credited with the discovery of a security
  vulnerability.

What is a GDB security bug?
===========================

  The primary use case of GDB is to diagnose issues in programs.
  These programs may be local, i.e. on the system on which GDB runs or
  remote, on a different system.

  A security issue is any unexpected behaviour that threatens the
  security of a system or network, or might compromise the security of
  data stored on it, while using GDB in a way that the maintainers
  have declared, in this document, is secure.

  Issues that arise in areas of GDB that the maintainers have declared
  as insecure, or when using GDB in a way that is noted as insecure
  within the document, should be reported as non-security bugs.  These
  bugs will still be accepted, and will be treated with the
  appropriate urgency.

  

  Whether a bug in GDB should be considered a security bug or not will
  depend on whether the area of GDB in which the bug occurs has been
  sufficiently hardened that the GDB maintainers feel able to declare
  that area of GDB secure.

  Where GDB has not been sufficiently hardened, then any bugs, even
  those that might appear to be security bugs, should be reported as
  non-security bugs; the expectation is that users should not be
  performing these actions outside of a sandboxed environment, and so
  no security threat exists.

  



  In the context of local debugging, any bugs in GDB that result in
  crossing of a privilege boundary are considered security bugs.  Some
  examples of crossing a privilege boundary include; being able to
  execute code as an arbitrarily different user, or accessing resources
  (e.g. files, sockets, etc) for which the original user would not
  normally have access.

  In the context of remote debugging, there are two components of
  relevance; GDB running on the local machine, and a remote target
  running as a potentially different user, with different permissions,
  on a remote machine.

  The GDB project provides one remote target, gdbserver.  As with GDB,
  any bugs in gdbserver that result in crossing of a privilege
  boundary are considered security bugs.

  Other projects also implement remote targets to which GDB can
  connect.  Any bugs in these remote targets are out of scope for the
  GDB project, and should be reported to the relevant project.

  The connection between GDB and a remote target is not protected by
  either authentication, or encryption.  Connecting to a remote target
  allows for arbitrary code execution on the remote system with the
  same privileges as the remote user, and any resource that the remote
  user can access can be read by GDB, and downloaded to the local
  machine on which GDB is running.  As such users need to take
  independent measures to secure the connection between GDB and the
  remote target.

What is a GDB security bug?
===========================

  The primary use case of GNU GDB is to diagnose issues in programs.
  These programs may be local, i.e. on the system on which GDB runs or
  remote, on a different system.

  A security bug is one that threatens the security of a system or
  network, or might compromise the security of data stored on it.  Any
  bug that facilitates the crossing of a privilege boundary can be
  considered a security bug.  Additionally, any bug that results in
  the unexpected execution of arbitrary code can be considered a
  security bug.

  In the context of local debugging, any bugs in GDB that result in
  crossing of a privilege boundary are considered security bugs.  Some
  examples of crossing a privilege boundary include; being able to
  execute code as an arbitrarily different user, or accessing
  resources (e.g. files, sockets, etc) for which the original user
  would not normally have access.  However, for the bug to be a GDB
  security issue, the privilege boundary crossing needs to be only
  possible, or significantly easier, when using GDB.  For example,
  using GDB to run an inferior that triggers a security bug in the OS
  kernel, or the wider OS environment is not a GDB bug, but should be
  reported to the relevant OS component.

  When GDB opens an executable for debugging, GDB will automatically
  process the symbols and debug information contained within the file.
  Some debug formats include domain specific languages which GDB must
  evaluate in order to correctly track program state.  Simply opening
  an executable within GDB for inspection, but not starting the
  executabe should not result in any untrusted code being executed
  that can effect changes outside of the GDB process.  Any bug that
  can trigger GDB to execute untrusted code without user interaction
  (either a direct command entered at a GDB prompt, a command entered
  from the command line, or a command entered into one of GDB's
  configuration files) would be a security bug.

  In the context of remote debugging, there are two components of
  relevance; GDB running on the local machine, and a remote target
  running as a potentially different user, with different permissions,
  on a remote machine.

  The GNU GDB project provides one remote target, gdbserver.  As with
  GDB, any bugs in gdbserver that result in crossing of a privilege
  boundary are considered security bugs.

  Other projects also implement remote targets to which GDB can
  connect.  Any bugs in these remote targets are out of scope for the
  GNU GDB project, and should be reported to the relevant project.

  The connection between GDB and a remote target is not protected by
  either authentication, or encryption.  Connecting to a remote target
  allows for arbitrary code execution on the remote system with the
  same privileges as the remote user, and any resource that the remote
  user can access can be read by GDB, and downloaded to the local
  machine on which GDB is running.  As such, users need to take
  independent measures to secure the connection between GDB and the
  remote target.

  If gdbserver is passed an executable when it is invoked, then
  gdbserver will automatically start the executable and then pause the
  executable at the first instruction waiting for GDB to connect.  Any
  bug that allows the debuggee executable to continue executing beyond
  the first instruction without an outside agent connecting to
  gdbserver would be considered a security bug.  Given the unsecure
  nature of the connection between GDB and gdbserver care should be
  taken if starting gdbserver with an untrusted binary to ensure that
  only a trusted GDB is allowed to connect and control the gdbserver
  process.

  While GDB is intended to be robust against untrusted binaries, it is
  not responsible for arbitrary code execution on a system.  As a
  result, any bugs exposed by untrusted binaries will be accepted and
  treated with appropriate urgency, but will not be considered
  security issues.  This applies to local as well as remote debugging;
  any bugs in gdbserver, exposed by untrusted binaries will be
  accepted and treated with appropriate urgency, but will not be
  considered security issues.

  Loading a binary into GDB should not result in any part of the
  loadable program being executed without a starting command being
  given to GDB (e.g. run, start, or starti).

  Loading a binary into gdbserver immediately creates a process
  running the given binary, which is stopped at the first instruction
  until GDB is attached.

  GDB assumes that the remote target is trusted.  While GDB is
  intended to be robust against incorrectly behaving remote targets, a
  malicious remote target could trigger bugs within GDB.  Any bugs
  exposed by connecting to an untrusted remote target will be accepted
  and treated with appropriate urgency, but will not be considered
  security issues unless the bug triggers a privilege boundary
  crossing.

  Care should be taken when creating tools that wrap around GDB in
  order to perform crash analysis, or binary analysis.  Such tools
  should only be run on trusted binaries, or should be run within a
  sandboxed environment in order to avoid the risks outlined above.
